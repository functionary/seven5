Seven5

Ian Smith, iansmith at acm dot org, GoSF, March 27 2013

Seven5 is named "Seven5" because names don't matter much. Google is a terrible name for a search engine company; no amount of good naming will save a bad project.
Seven5 is the prefix for the postal codes of Paris--in the city proper. Having a "75xxx" zip code is, for those of you old enough to remember, like having a 212-xxx-xxxx phone number.

Why

Not

In the interest of time, I can only really talk about why Seven5 is what it is and what it is not.
Defining web tools in opposition to something has worked well in the past. Rails, I argue, was defined in opposition to the horror that is (was?) J2EE. Camping and Flask are defined against what Rails and Django have become.

Slogans! Seven5 is...

Restful without remorse or pity.

Fiercely strong on typing.

Not much bothered about data storage.

Brutally reactionary towards the forces of dynamism.

Generally opposed to server side content generation.

... yet, strangely refreshing for web development.

I made up these quotes because I was inspired by Rob Pike's sense of brevity and pithiness.
  • REST is the only option
  • Tries to be strongly typed on both sides of the wire (Dart + optional typing)
  • Pushing towards "the single binary principle" -- no deps on filesystem. Want everything known at build time, understood behavior under load.
  • No template language, no way to generate web page on the server.
We pity the fools.

The World Is Different Now

The world has changed. Today's highly interactive applications a run almost entirely on the client side.
Client applications (browsers or otherwise) speak an protocol with the server. Content is generated on the client side based on interactions over the protocol (sometimes called API).
Seven5 has no way to generate server-side content. It helps you build a simple REST-based protocol that your clients can talk to.

Static Fans: Stand Up!

Watch Rob Pike's talk at Stanford about go (to a EE class). It sets up the argument that people have confused 'not sucking' with dynamic languages.
No real static option on the client side, ATM. Gollvm is clearly the right answer with asm.js.

You don't have a data problem.

Simple rule of thumb: If Ian hasn't heard of your company you don't have a data problem.
There are plenty of good choices for data storage. Fun ones include beedb, hood, modsql, etc.
If you are thinking about using a NO-SQL solution, read this slide again.
